## 0. Known bug fixed before this pass

**The legacy "Optimize CV" panel (`extension/src/sidepanel/optimize/index.js`) was leaking raw backend errors to the user.**

- **What was observed:** A failed `GENERATE_RESUME` call rendered as literal text on screen, e.g. `HTTP 500: {"error": {"code":"INTERNAL_SERVER_ERROR"...}}`.
- **Root cause:** `optimizeCv()` stored `res.error` (background/api.js's raw `"HTTP {status}: {response body}"` string) directly into `state.error`, and `render()` displayed it verbatim (`extension/src/sidepanel/optimize/index.js`, was line 196, now line ~201).
- **Why it mattered:** Violates `docs/pathfinder-uiux-requirements.md`'s Voice rule — "No raw error strings/JSON ever shown to users" — and is the exact bug already fixed once in `loading/index.js`/`detection/index.js` (the new Screen 3 flow), just never backported to this older code path.
- **Fix applied:** `optimize/index.js:201-202` now logs the real error via `console.error` and sets `state.error = "Try again."` (rendered alongside the existing `Message.GENERATE_FAILED` heading).
- **Regression test added:** `extension/tests/optimizeController.test.mjs` — `"generate failure: never shows the raw HTTP/JSON detail, real error still logged for debugging"`. Asserts the raw HTTP status and JSON body never appear in the DOM, and that the real error is still captured via `console.error` for debugging. The pre-existing `"generate failure: shows GENERATE_FAILED..."` test was updated to match the new copy (it had previously asserted the *buggy* raw-text output).
- **Status:** Fixed and verified. Full suite green (183 passed after this fix, see §7).

## 1. Title/company display gap on known-ATS pages

**What was tested:** Traced the exact code path from job detection through to the title/company text rendered on the Known ATS screen (using Ashby as the reported example).

**What I found — this is NOT one bug, it's three distinct possible causes, and I could not determine from code alone which one the user hit:**

**(A) Expected fallback — no analysis run yet for this tab.** `currentPageBlock()` in `extension/src/sidepanel/detection/index.js:199-213` renders:
```js
title.textContent = analysis?.title || (state === "known-ats" ? "Job listing detected" : "Possible job listing");
```
where `analysis` comes from `freshJobAnalysis()` (`detection/index.js:180-187`), which returns `null` unless a job analysis already exists **for this exact tab, with a URL matching the current page**. Job analysis only exists after the user clicks "Analyse this page" (`job-analysis/index.js`) — detection alone (the badge/confidence/atsName) never populates title or company. This is documented behavior, not a bug: the code comment at `detection/index.js:189-192` states this explicitly. **This is the most likely explanation if the screenshot/observation was taken before clicking "Analyse this page" on that tab.**

I confirmed this fallback path is *not* broken when analysis genuinely exists: `extension/tests/detectionController.test.mjs` has a passing test, `"known ATS: matching job analysis populates real title/company"`, which mounts a fresh, URL-matching job analysis and confirms the real title/company render. So the display logic itself is not the defect, if (A) is what happened.

**(B) A successful backend analysis can legitimately return a null title/company.** `backend/schemas/jobs.py:15-16` and `:38-39` both declare `title: str | None` / `company: str | None`. The LLM prompt itself (`backend/services/job_analysis_agent.py:39-40`) explicitly instructs `"title": string|null, "company": string|null` — the model is told null is an acceptable answer if it can't confidently extract these fields. If this happened for the Ashby page in question, the frontend's fallback-to-generic-copy behavior is *correct*, but the root cause would be LLM extraction reliability — **backend/prompt work, not a frontend bug.** I have no way to confirm from here whether this actually happened for the specific page observed; it needs a live reproduction with the network response inspected (or backend logs, as we did for the earlier 503 investigation).

**(C) A same-tab URL mismatch (theoretical, unconfirmed).** Both `detection.url` (`detect.js`, `location.href` at detection time) and the stored job analysis's `url` (`scrape.js:437`, `location.href` at scrape time) are captured independently, at different moments, and compared with strict equality in `freshJobAnalysis()`. In the normal flow these should always match — I found no evidence this is actually happening, and could not construct a realistic scenario where it would diverge given SPA route changes already re-trigger detection (`content/index.js:62-69`) before a user would click Analyse. I flag it only because it's a structurally possible failure mode, not because I found any evidence of it.

**Why it matters:** If it's (A), there's nothing to fix — this is working as designed, and the real gap is the *product experience* of not making it obvious to the user that they need to click "Analyse this page" first to see a real title. If it's (B), it's a backend/LLM quality issue outside this scope. I did not touch any code for this item — it needs a live reproduction to disambiguate, specifically: **click "Analyse this page" on that exact Ashby tab, then check whether the title populates.** If it does, this was (A). If it still shows generic text, check the network response for that `ANALYZE_JOB` call to see if `title`/`company` came back null (→ B) or check whether `GET_JOB_ANALYSIS`'s stored `url` matches `detection.url` exactly (→ C).

## 2. Known-ATS detection accuracy against real platforms

**What was tested:** Fetched real, currently-live job postings via `curl` (not `WebFetch`, which converts to markdown and loses DOM structure) from Greenhouse, Lever, Ashby, and Workday, and ran them through the actual `detect()` logic via the same JSDOM harness the test suite uses.

**Important methodology caveat, read before the numbers below:** `curl` only retrieves the server's initial HTML response — it does **not** execute JavaScript. Several of these platforms render their actual job content client-side after page load. The real extension's content script does *not* have this limitation: it runs `startObserving(rescore)` (`content/index.js:61`), a debounced `MutationObserver` (`observe.js`) that re-runs detection 400ms after DOM mutations settle — specifically built to catch async/SPA-rendered content. I confirmed this mechanism exists and is wired up, and it's thoroughly unit-tested (`observe.test.mjs`, `index.test.mjs`) — but I have no way to confirm, without a live browser, that it actually recovers full signal on these specific pages after rendering completes. **Treat the numbers below as a floor (worst-case, pre-render), not a ceiling.**

| Platform | Real URL fetched | `isJobPage` | `confidence` | Signals fired | Notes |
|---|---|---|---|---|---|
| **Lever** | `jobs.lever.co/loopreturns/...` | `true` | **1.0** | urlMatch, jsonLd, jobKeywords | Full signal — real title in raw HTML (`<title>Loop - Software Engineer</title>`), real JSON-LD, real headings. This one platform's server-rendered HTML is fully representative. |
| **Ashby** | `jobs.ashbyhq.com/notion/...` | `true` | **1.0** | urlMatch, jsonLd | Real title present, but `jobKeywords` (headings/body text) is `false` — visible job content is client-rendered; only the URL match and embedded JSON-LD (likely for SEO) saved full detection. |
| **Workday** | `cengage.wd5.myworkdayjobs.com/...` | `true` | **1.0** | urlMatch, jsonLd | Raw HTML has an **empty `<title>`** and **zero heading elements** (fully client-rendered SPA shell) — detection is saved entirely by the URL pattern match plus an embedded JobPosting JSON-LD blob. Confirms JSON-LD is a genuinely load-bearing signal for this platform, not a nice-to-have. |
| **Greenhouse** | `boards.greenhouse.io/reddit/jobs/5650188` | `true` | **0.6** | urlMatch only | Raw `<title>` is generic ("Jobs at Reddit", not the specific role) and no JSON-LD/headings fired in the static fetch — same client-rendered-shell pattern as Ashby/Workday, but **without** a JSON-LD fallback to compensate. Confidence rests entirely on the URL pattern (`ATS_URL_PATTERNS`, `detect.js:16-17`). |

**Why it matters:** `isJobPage` came back `true` for all four in this worst-case test, which is a good sign — the URL-pattern allowlist alone is carrying real weight and none of these would be *missed* as job pages. But confidence varies from 0.6 to 1.0 purely due to how much of each platform's content is server- vs. client-rendered, and the *title/company display* (§1) depends on the backend's LLM analysis of scraped text, which I did **not** run against these four real pages (that requires the live scrape→`ANALYZE_JOB`→LLM pipeline per page, real API cost — out of scope for this pass, see §6 for what *was* run end-to-end).

**Follow-up attempt: does the `MutationObserver` rescore (`content/index.js:61`) actually recover full signal on Greenhouse after client-side rendering completes?** I tried to check this directly rather than leave it as an open question, using `jsdom`'s `runScripts: "dangerously"` mode (real script execution, not just static HTML parsing) against the live Greenhouse URL, waiting 8.7s to approximate render time. Result: **partial, inconclusive.** Real rendering did happen — headings went from 1 (static fetch) to 8, and body text grew from near-empty to ~39KB — but `document.title` still never updated to the specific job title (stayed "Jobs at Reddit"), and the rendered body text showed CSS-in-JS style declarations leaking into `textContent`, a known `jsdom`-vs-real-browser fidelity gap for apps using styled-components/emotion. A follow-up attempt to also compute `detect()`'s confidence against this rendered DOM hit a DNS resolution failure on the same host — most likely Greenhouse's infrastructure rate-limiting or flagging the unusually heavy resource-fetching pattern `jsdom`'s resource loader produces (very different traffic shape than a normal browser or a single `curl`), though I can't confirm that's the specific cause.

**Bottom line on this specific question: not reproducible reliably from this environment.** `jsdom` script execution is not a real browser — it has genuine, demonstrated fidelity gaps (the CSS-leak artifact) and in this case couldn't even complete a second, cleaner attempt. This is a **live-browser-only verification item**: load the extension, visit a real Greenhouse job listing, and check the side panel's confidence/badge state after the page finishes loading. I'm flagging this explicitly rather than leaving the original claim (JSON-LD/headings alone determine confidence) unverified by implication.

## 3. Unknown-ATS and keywords-only states against real content

**What was tested:** A real DoorDash job posting (`careersatdoordash.com`, custom domain, not matching any `ATS_URL_PATTERNS` entry — a legitimate Unknown-ATS candidate) and a real career-advice article (`dailyremote.com/advice/remote-work-statistics-2026`) as a keywords-only candidate.

**What I observed:**

- **DoorDash (intended Unknown-ATS test): `confidence: 0`, not even keywords-only.** The raw HTML is a fully client-rendered SPA shell — generic `<title>DoorDash</title>`, 2 near-content-free heading elements, no JSON-LD. This means my one real Unknown-ATS candidate never actually reached the Unknown ATS *or* Keywords-only state in this test — it landed in "no signal at all." **I could not verify the Unknown ATS screen's copy against a real live page this session.** The copy-accuracy claim rests entirely on the existing synthetic unit test, `detectionController.test.mjs`'s `"unknown ATS: screen shown, 2 bars filled, missing-signals copy names what's absent"` — which I did verify passes, and which does exercise real logic: `missingSignalsCopy()` (`detection/index.js:169-178`) dynamically builds its text from `Object.entries(MISSING_SIGNAL_LABELS).filter(([key]) => !signals[key])` — i.e. it genuinely reflects which signals are actually missing on that page, not hardcoded placeholder text. That's solid *unit* evidence; it is not the same as live-page evidence.

- **dailyremote.com article (keywords-only): `confidence: 0.15`, `isJobPage: false`.** `jobKeywords: true` (from `hasJobBodyContent()` — the article discusses salary/compensation/benefits stats), `jobHeadings: false`, `metaTags: false`. Correctly stays below the 0.4 threshold, so it does **not** offer tailoring — but it does still register nonzero confidence and would show the "Keywords only: Not enough to tailor from" screen on a page that is not job-search-related to the user at all (a statistics article). This is a real, live-confirmed instance of the residual false-positive class flagged in §4/the earlier session's `NON_JOB_HOSTS` comment in `detect.js` — low severity (no tailoring offered, matches the "don't tailor from thin data" design rule) but not fully closed.

**Why it matters:** The Unknown-ATS "names what's missing" requirement is well-covered at the unit level but effectively untested against real content this session, purely because my one live candidate scored zero rather than landing in that state. This is worth a follow-up with a better real Unknown-ATS candidate (a smaller company's server-rendered careers page, not a big-tech SPA).

## 7. Full automated test suite

**Result: 183 passed, 0 failed**, run via `npm test` in `extension/`, immediately after the fix in §0.

**Test-quality audit** (per the request not to just report green):

- **`fixture-replay.test.mjs`'s positive-fixture loop is weaker than it looks.** For each of the 7 "true positive" fixtures (`greenhouse.html`, `lever.html`, `workday.html`, `linkedin.html`, `indeed.html`, `generic-jsonld.html`, `generic-no-ats.html`), the test only asserts `d.isJobPage === true` (`fixture-replay.test.mjs:130-134`) — it does not check *which* signals fired or the confidence value. A future change that broke one signal while an unrelated signal happened to compensate (e.g. `jsonLd` silently stops working but `jobHeadings`+`jobBody` still cross 0.4) would pass this suite without detection. Not wrong, just less precise than it could be. Worth tightening if this pipeline gets more investment.
- **The suite's own documentation already flags its limits, and this pass confirms why that matters.** `fixture-replay.test.mjs:1-18`'s header comment states fixtures are hand-built, not scraped from live sites, specifically because real markup drifts and goes stale. That's a reasonable engineering tradeoff for a fast, deterministic CI suite — but it also means the existing automated suite was **never** sufficient evidence for "does this work against real platforms" claims (§2's ask). The live-`curl` testing in this report is a genuinely new layer of verification, not a duplicate of existing coverage.
- **No `assert(true)` or similarly vacuous assertions found** in a grep across all 21 test files.
- **SPA-navigation and context-invalidation coverage (`index.test.mjs`, `observe.test.mjs`) is unusually thorough** for this codebase — adversarial/edge-case tests (rapid route changes after context invalidation, mid-session extension reload) exist specifically because a prior bug was found there. Good precedent; the newer `detection`/`loading` test files (`detectionController.test.mjs`, `loadingController.test.mjs`) follow the same real-behavior-not-implementation-detail testing style established there and in `idleController.test.mjs`.
